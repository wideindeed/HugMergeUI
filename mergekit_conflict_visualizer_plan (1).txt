-------------------------------------------------------------------
1. THE ORIGINAL PITCH (for reference)
-------------------------------------------------------------------
Build a visual tool on top of mergekit (the open-source LLM merging
library by Arcee AI). Instead of hand-writing YAML and hoping the
merge isn't "brain-dead," users would:
  - Load/build a merge config visually
  - See layers as a stacked diagram, color-coded by which parent
    model contributed what (e.g. SLERP blend ratios as bar splits)
  - See passthrough/"frankenmerge" slicing drawn as literal
    stitched connections between input and output layers
  - Get static sanity checks against the Hugging Face API
    (layer count mismatches, hidden dim mismatches, vocab size
    mismatches)

PROBLEM: A near-identical product already exists — "MergeKit Hub"
(mergekit.com) — offering a config generator + an interactive
"Merge Map" visualizer with the same layer-level lineage view.
Building this as described = shipping a clone.


-------------------------------------------------------------------
2. THE NEW IDEA: Predictive Conflict Diagnostics
-------------------------------------------------------------------
Keep the visualizer shell, but make the CORE FEATURE something the
competitor doesn't have: telling the user whether a merge will
actually be good BEFORE they spend hours running it on a GPU.

Two layers of this, in order of ambition:

  A) WEIGHT-SPACE CONFLICT SCORE (build this first)
     For each layer (or each parameter tensor), compute:
       - Sign conflict rate: % of parameters where Model A's
         update direction disagrees with Model B's update
         direction, relative to the base model
       - Redundant magnitude: how much of each model's change is
         "noise" that TIES-style merging would prune anyway
     Surface this as a per-layer heatmap laid over the same
     stacked-layer diagram from the original pitch:
       "Layer 14: 62% sign conflict — expect quality loss here"
     This is pure tensor math on downloaded (or streamed) weights.
     No GPU inference required. No full merge required.

  B) CHEAP PROXY EVALUATION (stretch goal, phase 3+)
     Run a small batch of held-out prompts through both parent
     models, capture hidden states layer-by-layer, and simulate
     what the chosen merge method (SLERP/linear/TIES) would do to
     those activations — without materializing the full merged
     model. Show projected output divergence per layer. This is
     the "pre-flight check" mergekit itself doesn't offer.

Why this is the differentiator:
  - It's predictive, not just descriptive. Their tool shows what
    you configured. This tool shows whether it's a good idea.
  - It's hard to bolt on after the fact — it requires real tensor
    math, not just prettier D3 rendering.
  - It directly answers the #1 complaint about mergekit today:
    "I waited 3 hours and the merge was bad."

4. STEP-BY-STEP BUILD ORDER
-------------------------------------------------------------------
PHASE 0 — Setup (few days)
  WHAT: Environment setup, repo scaffolding
  WHERE: Local machines, no GPU needed for this phase
  - pip install mergekit, huggingface_hub, safetensors, torch (CPU)
  - Set up a monorepo: /backend (FastAPI or Flask), /frontend (Vite
    + React + TypeScript)
  - Pick 2-3 small, freely-downloadable model pairs to test against
    (e.g. small Mistral/Llama variants under 3B params — keeps
    laptop-friendly download sizes and compute)

PHASE 1 — Config Parser (Easy, ~1 week)
  WHAT: Parse mergekit YAML -> clean JSON tree
  WHERE: Backend
  - Handle merge_method, slices, models, base_model, parameters
  - Unit test against the example configs in mergekit's own
    /examples folder on GitHub (real-world configs = free test data)

PHASE 2 — Static Architecture Sanity Checks (Easy-Medium, ~1 week)
  WHAT: Hugging Face API calls to fetch config.json for each
        parent model referenced in the YAML
  WHERE: Backend
  - Compare num_hidden_layers, hidden_size, vocab_size
  - Return structured warnings (not just strings) so frontend can
    render them as flagged nodes on the diagram

PHASE 3 — Conflict Score Engine (Medium-Hard, ~2-3 weeks)
  WHAT: The actual differentiator — sign conflict + redundancy
        metrics per layer, computed from safetensors weight diffs
  WHERE: Backend
  - Start with 2-model linear/SLERP case only — don't try to
    support every merge method on day one
  - Reference: TIES-Merging paper (Yadav et al.) for the sign-
    conflict and redundant-parameter formulas
  - Output: per-layer JSON score (0.0-1.0 conflict, 0.0-1.0
    redundancy) that the frontend can map to color intensity

PHASE 4 — Visual Diagram + Heatmap (Medium, ~2-3 weeks, parallel
           with Phase 3 once JSON shape is agreed on)
  WHAT: Stacked layer diagram, color-coded by contribution %,
        heatmap overlay from Phase 3 scores, passthrough slicing
        shown as connecting lines
  WHERE: Frontend
  - React Flow for node/edge layout, or D3 if you want full custom
    control over the heatmap rendering
  - Click a layer -> side panel with exact conflict/redundancy
    numbers and a plain-English note

PHASE 5 — Proxy Evaluation (Stretch goal, later)
  WHAT: Small-batch hidden-state capture + divergence estimate
  WHERE: Backend, needs at least a consumer GPU (8GB+ VRAM) or a
         rented cloud GPU for testing at this stage
  - Only attempt this once Phases 1-4 are solid and shipped

8. ENVIRONMENT GOTCHAS (learned the hard way, first hands-on run)
-------------------------------------------------------------------
  - PyTorch install: `pip install torch` alone can silently grab the
    CPU-only build. Always install explicitly via the CUDA index,
    e.g. `pip install torch --index-url
    https://download.pytorch.org/whl/cu126`. Verify with:
      python -c "import torch; print(torch.cuda.is_available(),
      torch.cuda.get_device_name(0))"
  - `nvidia-smi`'s "CUDA Version" field shows the MAX version your
    driver supports, not what you must install. Drivers are
    backward-compatible — a stable, well-tested CUDA build (12.6 or
    13.0) is safer than chasing the newest experimental/nightly one,
    since libraries like transformers/accelerate/mergekit lag behind
    on bleeding-edge CUDA support.
  - Python version: stick to 3.11 or 3.12 for the venv. Very new
    Python releases (3.14 etc.) can have shaky compatibility with
    transformers/accelerate/torch before the ecosystem catches up.
  - PyCharm interpreter mismatch: if a script errors on a package
    you KNOW is installed (e.g. "accelerate not found"), check the
    traceback file paths. If they point to a global Python install
    instead of your venv's site-packages, PyCharm's run
    configuration/project interpreter isn't set to the venv. Fix in
    Settings > Project > Python Interpreter, or just run from an
    activated terminal (`myenv\Scripts\activate`) to sanity-check.
  - Terminal word-wrapping is not a real bug: PowerShell/console
    windows can visually break a word across lines when wrapping
    output. The generated text itself isn't corrupted — pipe to a
    file (`> output.txt`) or view in an editor to confirm.

9. CLARIFICATION: WHAT SLERP ACTUALLY DOES (for the diagram design)
-------------------------------------------------------------------
Important for Phase 4's visual design — SLERP is NOT slicing/
stitching (that's passthrough/frankenmerge) and NOT a layer
criss-cross. At every matching layer position, it spherically
interpolates between Model A's and Model B's weights for that same
layer, based on a blend ratio `t` that can vary by depth:

    new_layer_N = spherical_blend(A_layer_N, B_layer_N, t=value_at_N)

Every output layer contains contributions from BOTH parents unless
t=0 or t=1 exactly at that depth. "Spherical" (vs. simple averaging)
matters because naively averaging two high-dimensional weight
vectors can shrink their magnitude and lose information; SLERP
interpolates along the arc between them instead, preserving norm.

Design implication: the per-layer diagram should show a genuine
blend gradient (e.g. a 70/30 split bar), NOT two discrete source
blocks feeding into one output block. Save the "two discrete
blocks stitched together" visual specifically for passthrough/
frankenmerge configs, where that representation is accurate.
